## 메타인지 하기

### 1. 현재 TypeScript 경험

Q. TS를 써본 적이 있는가? 있다면 어느 정도?

A. TS를 써본 적이 있다. 다만 단순히 React와 next에서 각 컴포넌트와 인자의 타입 검증을 위한 용도로만 뒤늦게 추가해서 썼어서, 타입스크립트로 처음부터 프로젝트를 설계하고 여러 키워드와 기능을 사용해본 적은 없다.

### 2. 기존에 새로운 개념을 학습할 때 어떻게 했나요? (수업시간에 v1 떠올렸던 것 첨부)
- 카카오톡 챗봇 개발

1. 챗봇 개발 플랫폼에서 여러 기능을 직접 눌러가며 어떤 것들이 있는지 살펴보고 예상한다.
2. 블로그나 유튜브에서 기본적인 사용 방법을 찾아 본다.
→ 자료가 거의 없음
3. AI에게 기본 사용법을 물어본다.
→ 데이터가 거의 없어서 잘 모름 + 이상한 답변 함
4. 공식 문서를 찾아본다.
5. 공식 문서의 내용들을 정독하고 직접 따라해보면서 사용 방법과 작동 원리를 익힌다.
6. 직접 해보며 막히는 부분에 대해서 (ex) 백엔드 연동) 다시 2~5번을 반복한다.
7. 직접 원하는 기능을 구현해보며 시행 착오를 겪고, 문제를 해결한다.

## 학습 로그 #1

**시간**: 03/24 15:00 ~ 16:30 (약 90분)

**학습 범위**: (typescript-exercises #1~#5)

### 1. 첫 학습 전략 설계

- 우선 무작정 문제를 풀려고 달려들기보다는, 문제를 보고 이 문제의 의도가 무엇이고, 이 문제로부터 어떤 타입스크립트 지식을 얻을 수 있는지를 파악했습니다.
- 그렇게 되니 자연스럽게 문제 해결 과정이 머릿속에 정리가 되고, 제가 해야 하는 건 단순히 해당 풀이 순서대로 코드로 옮기는 방법을 학습하기만 하면 됐던 것 같습니다.

따라서 문제 풀이 전략(학습 전략)의 순서는 다음과 같았습니다.

1. 문제 의도 파악
a) 문제 상단의 Intro, Exercise 영어 설명 자세히 읽기
b) 코드에서 에러 뜨는 부분 확인 후 왜 에러인지 유추 해보기
    
2. 문제 해결
a) 아는 문법으로 풀 수 있고 쉬운 문제라면 바로 해결, AI 피드백 X
b) 모르겠으면 코드 하단에 참고 링크 참고하여 문제 해결, AI 피드백 X
    
exercise 5번까지는 위 2가지로 모두 문제가 풀려서, 문제가 풀리지 않은 경우에 대해선 다음 로그에서 작성해 볼 예정입니다.

### 2. 전략 평가
효과적이었던 것과 그 이유

-가장 효과적이었던 점은 문제를 바로 풀지 않고, 먼저 문제 의도와 에러 원인을 파악한 것입니다.
덕분에 풀이 방향을 먼저 정리한 뒤 코드를 작성할 수 있어서 덜 막막했습니다.

-또 아는 문제는 스스로 풀고, 모를 때만 참고 링크를 본 점도 효과적이었습니다.
이 과정에서 내가 아는 것과 모르는 것을 구분할 수 있었고, 부족한 부분만 보완할 수 있었습니다.

비효과적이었던 것과 그 이유

-이번에는 큰 문제가 없었지만, 앞으로 난이도가 올라가면 문제 의도 파악 단계에서 오래 막힐 것 같습니다.
그래서 다음에는 오래 붙잡기보다, 어디까지 알고 어디서부터 모르는지 구분하는 방식이 필요할 것 같습니다.

### 3. AI 피드백

```
전체적으로 좋은 전략이었습니다.

문제를 보자마자 풀지 않고, 먼저 문제 의도와 에러 원인을 파악한 뒤 해결하려고 한 점이 특히 좋았습니다.

이 방식은 단순히 정답만 맞히는 게 아니라, 이 문제로 어떤 타입스크립트 개념을 배우는지 생각하게 해줘서 학습 효과가 높습니다.

다만 아쉬운 점은, 문제를 푼 뒤에 배운 개념을 명확하게 정리하는 단계가 부족했다는 것입니다.

그래서 풀 때는 이해한 것 같아도, 나중에 비슷한 문제를 다시 만나면 기억이 흐려질 수 있습니다.

### 유용한 제안 1가지

문제를 푼 뒤, “이 문제로 배운 타입스크립트 개념 1줄”을 꼭 남기기

예:

- 객체 타입 구조를 맞추는 문제였다.
- 함수 인자 타입과 반환 타입의 관계를 이해해야 했다.
- 값의 형태와 타입 선언이 일치해야 한다는 점을 배웠다.

이렇게 한 줄씩 남기면, 문제 풀이가 그냥 경험으로 끝나지 않고 재사용 가능한 지식으로 남습니다.
```

### 4. 다음 타임에 바꿀 것

유지할 것과 그 이유
- 문제를 바로 풀지 않고, 먼저 문제 의도와 에러 원인을 파악하는 방식은 유지할 것입니다.
- 이 방법이 문제를 더 구조적으로 이해하게 해줬고, 풀이 방향을 잡는 데 도움이 되었기 때문입니다.
    
바꿀 것과 그 이유
- 문제를 푼 뒤에는 배운 타입스크립트 개념을 한 줄로 정리하는 과정을 추가할 것입니다.
- 그래야 문제를 푸는 데서 끝나지 않고, 다음 문제에도 활용할 수 있는 지식으로 남기 쉽기 때문입니다.

### 1. 이번 타임의 학습 전략

예상대로 난이도가 높아지니 AI 도움 없이, 문서만 보고 풀었을 때 막히는 부분이 생겼습니다. 따라서 아래의 방법을 적용해봤습니다.

1. 문제를 풀며 막혔을 때
- 문제를 풀며 막히는 지점이 생겼을 때 AI를 활용
- 다만 단순히 문제를 전부 주며 AI에게 답을 요구하는 것이 아니라, 내가 현재 어느 부분에서 막혔고, 막힌 부분 관련해서 사용할 수 있는 문법이 어떤 것이 있는지 물어보는 형식으로 질문하여 저 스스로 한 번 더 사고할 수 있는 기회를 주었습니다.

- 이미 타입스크립트에 대해 공부한 적이 있었기에, 이미 아는 문법을 활용한 문제도 나왔습니다.
- 그럼에도 1번과 같은 사이클을 반복하며 새로운 개념을 배우듯이 학습했을 때, 제가 아는 개념에서도 몰랐던 부분이나 새로운 활용법을 알 수 있었습니다.

## 학습 로그 #3

**시간**: 03/25 16:00 ~ 18:00 (약 120분)
**학습 범위**: (typescript-exercises #10)

### 1. 이번 타임의 학습 전략
- AI 도구를 적극적으로 사용했습니다.
- 용도가 다른 3가지의 gems를 제작하여 문제 풀이와 학습에 도움을 받았습니다.
1. Typescript exercise Helper
	- typescript exercise 문제와 정답을 가지고 있으며, 해결 방향성을 제시해주고 '문제 풀이'에 도움이 되는 gems
2. Typescript 샌드백
	- 타입스크립트 문법이나 사용에 대해 궁금증이 생겼을 때 지속적인 꼬리 질문과 토론을 통해 사고력을 늘리고 지식을 얻을 수 있는 gems
3. Typescript Study Helper
	- 공부를 하며 딥다이브하고 싶은 개념이나, 기록하고 싶은 문제점 등을 제시하면 지정한 기술 블로그 form으로 답변해주는 gems

- exercise #10 문제는 여러 제너릭 개념이 쓰이고, 특히 bonus exercise가 상당히 어려웠기에 천천히 스텝 별로 이해하려고 노력했습니다.
- 결론적으론 스스로 문제 해결을 하지 못하여 gems의 도움을 받아 문제를 풀어보고, 혼자서 처음부터 풀 수 있을 정도로 이해하기 위해 여러 번 풀어봤습니다.

### 2. 전략 평가
효과적이었던 것과 그 이유
- 여러 gems를 만드니 상황에 따라 각 역할에 맞는 gems와 대화를 할 수 있다는 점에서 효과적인 학습이었습니다.
- 특히 Typescript 샌드백은 저로 하여금 더 많은 사고를 할 수 있게 만들어주었던 것 같습니다.

비효과적이었던 것과 그 이유
- gemini gems를 사용하니 '빠른 모드', '사고 모드', 'pro 모드'가 있는데 확실히 [빠른 모드]는 속도는 빠르지만 답변의 깊이가 낮았고, [사고 모드]로 계속 대화를 하니 시간이 오래 걸려 효율적이지 못했던 것 같습니다.
- typescript 문법 뿐만 아니라 다른 js 기본 개념(promise, callbak 등)도 쓰여서 해당 부분도 재학습하는 부분에서 시간이 많이 쓰였던 것 같습니다.

### 3. AI 피드백

```
1. 전반적인 피드백

이번 학습은 전체적으로 방향이 좋았습니다.
특히 인상적인 점은, AI를 하나로 뭉뚱그려 쓰지 않고 문제 풀이용 / 토론용 / 기록용으로 나눠서 사용했다는 것입니다. 이 방식은 생각보다 중요합니다. 왜냐하면 “지금 내가 필요한 게 정답 방향인지, 개념 토론인지, 학습 기록인지”를 분리해주기 때문에 AI 사용이 훨씬 덜 수동적이기 때문입니다.

또한 어려운 제너릭 문제에서 바로 포기하지 않고, 도움을 받아서라도 여러 번 다시 풀어보며 이해하려고 한 태도도 좋았습니다. 이번 타임은 “스스로 완전 해결”은 못했지만, 그 대신 어디서 막히는지, 어떤 개념이 부족한지, 어떤 도구가 내 사고를 더 잘 끌어내는지를 꽤 정확하게 파악한 시간이었습니다. 이건 실제로 다음 타임 효율을 높여주는 의미 있는 성과입니다.

다만 한 가지 아쉬운 점은, 이번 학습이 ‘이해한 것’과 ‘혼자 재구성할 수 있는 것’ 사이의 차이를 조금 남겨둔 채 끝났다는 점입니다. gems의 도움으로 이해한 뒤 다시 풀어본 것은 좋지만, 완전히 아무 도움 없이 처음부터 구조를 꺼내는 훈련까지 충분히 갔는지는 별개입니다. 다음 타임에는 이 간극을 줄이는 쪽으로 가면 좋겠습니다.

2. 유용한 제안 1가지

다음부터는 문제를 해결한 직후 **“무도움 재구성 10분”**을 꼭 넣어보세요.

방법은 단순합니다.

gem 대화창 닫기
정답 코드 안 보기
IDE 힌트 최소화
그 상태에서
문제 요구사항 1줄 요약
핵심 제너릭 관계 2~3줄 정리
정답 코드 다시 작성
막힌 지점 1줄 기록

이 과정을 넣으면 “아는 것 같은 상태”와 “진짜 내 것으로 된 상태”가 바로 구분됩니다.
```
