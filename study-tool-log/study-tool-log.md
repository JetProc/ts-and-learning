# 학습 도구 로그

- 학습에 도움이 되는 도구를 만들고, 기록합니다. 필요하면 코드를 남깁니다.

## **도구 이름**
- Typescript Exercise Helper

## **도구 유형** (예: GPTs, gems, Claude Code(skill, command, agent, ...) Chrome Extension, CLI, 웹사이트 등)
- gems

## **해결하려는 문제**: 어떤 학습 상의 불편/문제를 해결하려 했는가?
1. 개념과 실전 사이의 막막함
- "제네릭(Generics)을 공부했는데, 이 문제에서 어떻게 써야 할지 모르겠다"와 같이 이론은 알아도 문제 적용에 어려움을 겪는 학습자들에게 징검다리 역할을 해줄 가이드가 부족했습니다.

2. 학습 이탈 방지
- 혼자 문제를 풀다가 막히면 구글링이나 공식 문서의 방대한 양에 압도되어 학습을 포기하기 쉽습니다. 이때 적절한 수준의 힌트(개념 키워드 등)를 제공하여 학습 흐름을 유지하고자 했습니다.

3. 문제 의도 파악의 어려움
- typescript-exercises는 코드로만 구성되어 있어, 각 단계에서 '무엇을 테스트하려는지'에 대한 친절한 설명이 부족한 부분을 보완하려 했습니다.

## **어떻게 만들었는가**: 간단한 제작 과정
1. 학습 데이터 분석
- typescript-exercises 오픈소스 저장소의 전체 구조를 분석했습니다. 특히 각 문제의 index.ts(문제), test.ts(검증 로직), index.solution.ts(해답)를 대조하여 출제 의도를 파악했습니다.

2. 단계별 가이드 로직 설계
- 일반적인 AI처럼 정답을 바로 제시하지 않도록 **'방향성 → 키워드 → 힌트'**로 이어지는 3단계 가이드 시스템을 프롬프트 엔지니어링을 통해 구현했습니다.

3. 페르소나 설정
- 사용자가 스스로 정답을 찾아낼 수 있도록 격려하면서도, 기술적으로는 엄격하게 검증하는 '친절하지만 타협 없는 멘토'의 목소리를 입혔습니다.

4. 실시간 코드 참조
- 사용자가 문제 번호를 말하면 즉시 해당 디렉토리의 파일을 읽어 들여 가장 정확한 조언을 할 수 있도록 RAG(Retrieval-Augmented Generation) 기반의 환경을 구축했습니다.

## **어떻게 도움이 되었는가**: 실제 사용 경험과 효과
1. 자기주도 학습 강화
- 정답 코드를 복사 붙여넣기 하던 습관을 버리고, 제시된 방향성을 토대로 interface, generics, union type 등 핵심 개념을 직접 고민하게 유도합니다.

2. 학습 효율성 증대
- 너무 막막할 때는 '개념 키워드'를 통해 공식 문서를 찾아볼 지점을 알려주고, 결정적인 순간에 '힌트'를 제공하여 학습의 흐름이 끊기지 않게 돕습니다.

3. 실전 감각 익히기
- 실제 오픈소스 프로젝트 구조 내에서 문제를 풀어나가므로, 단순히 문법을 아는 것을 넘어 실제 개발 환경에서 TypeScript가 어떻게 에러를 잡아내는지 체감할 수 있습니다.

4. 심리적 안정감
- 혼자 풀 때 느끼는 막막함을 해소해주며, 마치 곁에서 코드 리뷰를 해주는 동료가 있는 것 같은 효과를 주어 완주율을 높여줍니다.



## **도구 이름**
- Typescript 샌드백

## **도구 유형** (예: GPTs, gems, Claude Code(skill, command, agent, ...) Chrome Extension, CLI, 웹사이트 등)
- gems

## **해결하려는 문제**: 어떤 학습 상의 불편/문제를 해결하려 했는가?
1. 일방적인 정답 제공으로 인한 사고 정체
- AI가 준 코드를 생각 없이 복사하기만 해서 정작 본인의 문제 해결 능력이 늘지 않는 '의존성 문제'를 해결하고자 했습니다.

2. 방대한 정보량에 의한 학습 피로도
- 한꺼번에 쏟아지는 장황한 설명 대신, 대화를 통해 꼭 필요한 만큼의 힌트만 단계적으로 제공하여 학습의 몰입도를 높이고자 했습니다.

3. 궁금한 점이 있을 때 물어볼 수 있는 상대 필요
- 궁금한 점을 단순히 검색해보는 것이 아닌, 저만의 사고 방식을 검증해줄 멘토의 역할을 수행할 gem이 필요했습니다.

## **어떻게 만들었는가**: 간단한 제작 과정
1. 페르소나 및 역할 정의
- 단순 답변기가 아닌 '토론 파트너'로 정의하고, 꼬리에 꼬리를 무는 질문을 하도록 유도했습니다.

2. 실전 데이터 기반의 컨텍스트 주입
- TypeScript Exercise 레포지토리를 학습시켜 이론적인 설명뿐만 아니라 지금 하고 있는 고민과 문제를 엮어서 대답할 수 있도록 했습니다.

3. 상호작용 루프 강제화
- 모든 답변 끝에 반드시 '역질문'을 던지게 하여 사용자가 대화의 주도권을 쥐고 계속 사고하게끔 구조화했습니다.

## **어떻게 도움이 되었는가**: 실제 사용 경험과 효과
1 .능동적 사고를 유도
- 사용자가 자신의 코드를 AI에게 직접 설명하게 함으로써, 머릿속에 흩어져 있던 개념을 스스로 논리적으로 정리하는 계기를 제공합니다.

2. 소크라테스식 대화법을 통한 근본 원인 파악
- 단순히 에러를 고치는 수준을 넘어 "왜 이 타입이 필요한가?"를 끊임없이 묻고 답하며 TypeScript의 핵심 원리를 체득하게 합니다.

3. 사고의 한계를 확장하는 날카로운 역질문
- 답변 뒤에 이어지는 역질문이 사용자로 하여금 미처 생각하지 못한 예외 케이스나 타입의 확장성까지 고려하게 만들어 학습의 깊이를 더합니다.
